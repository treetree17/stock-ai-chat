# 项目报告：智能股票投资 AI 助手

## 1. 引言（Introduction）

### 1.1 项目构想 (Project Idea)
随着金融市场的日益复杂化，个人投资者在面对海量的市场数据、实时新闻以及冗长的企业财报时，往往难以快速提取有价值的信息并做出投资决策。为了解决这一痛点，我们构想了一个基于大语言模型 (LLM) 的“智能股票投资 AI 助手”。该助手旨在模拟专业金融分析师的工作流程，结合实时的量化行情、最新的互联网财报新闻以及深度本地研报，为用户提供一站式、自然语言交互的智能投资分析咨询服务。

### 1.2 解决方案的关键组成部分 (Key Components of the Solution)
本系统的解决方案主要由以下几个核心技术组件构成：
1. **意图识别与实体路由**：自动识别用户的提问意图（判断是否为股票金融相关问题），并精准提取股票实体（如“阿里”转换为“9988.HK”）。
2. **多源数据并发抓取网络**：
   - **量化行情源 (YFinance)**：获取股票当前的实时价格、开盘价、涨跌幅及 K 线图表配置。
   - **实时外部资讯 (Google News RSS)**：在用户提问瞬间，在线抓取最核心、最新发生的几条全球新闻短讯。
   - **本地深度化知识库 (RAG Vector DB)**：通过 SQLite 存储预先处理好的深度企业财报和文章的向量，系统根据问题运用余弦相似度检索深度背景。
3. **大模型推理引擎 (LLM Service)**：整合上述所有的上下文信息，构建高度定制化的 Prompt，指导大语言模型流式输出逻辑严密的金融分析。

### 1.3 主要成果总结 (Summary of Key Findings)
通过本项目的开发与测试，我们成功构建了一个轻量级的 MVP（最小可行性产品）。系统能够在 5 秒内完成从“意图识别”到“多源并发检索”再到“大模型结合研报与实时新闻生成解答”的全过程。测试表明，通过引入 RAG（检索增强生成）机制与实时联网查询，模型彻底摆脱了“幻觉”与“知识库过期”的限制，不仅能精准作答近期的突发财报事件，还能提取本地导入的专业长文给出深度见解，显著提升了垂直领域问答的质量。

## 2. 方法论（Methodology）

### 2.1 解决业务问题的方法 (How the Business Problem is Solved)
本系统采用的是“多智能体协作与 RAG 融合架构”来解决投资者面临的信息过载问题。传统的 LLM 具有信息滞后和缺乏具体企业私密研报的问题，我们通过设计以下管线（Data Flow）来解决：

1. **动态意图拦截机制**：系统首先对用户输入进行轻量级的分析，如果在本地意图词库匹配到该问题并非股票金融相关（如普通的日常问候 "hello" 或常识问答），则绕过复杂的金融数据查询流程直接进入基础对话模式，极大地节省了系统资源与 API 成本。
2. **并发式数据流体系 (Concurrent Data Pipeline)**：当检测到明确的股票金融业务意图时，后端 (FastAPI) 利用 Python 的 `asyncio` 实现多路并发数据采集：
   - 同步通过 YFinance API 获取客观交易数字；
   - 同步触发 `LiveNewsService` 进行时事新闻的轻量化爬取；
   - 同步将用户查询词通过本地 FastEmbed 模型计算 Embedding 向量，进入本地知识库 (SQLite) 做近似近邻搜索 (KNN/Cosine Similarity)，找出预摄入 (Ingest) 的财报段落。
3. **Prompt 组装与情景注入**：将抓取到的短讯、研报段落、股票基础面数据以及聊天历史对话，无缝拼接为“系统提示词 (System Prompt)”，迫使 LLM 扮演金融分析师的角色进行阅读理解与综合归纳。

### 2.2 核心挑战与克服策略 (Major Challenges and Strategies)

**挑战一：大模型信息滞后与“幻觉”现象**
纯粹依赖原生大参数模型提供的回答，容易生成已经过期（比如一年前）或者凭空捏造的股价信息，这对金融场景是致命的。
*克服策略*：引入两套新闻机制体系。对于最新的几小时内的新闻，我们利用 `LiveNewsService` 突破无网络限制，从外部实时抓取；对于长线深度认知，如企业年报，我们离线开发了 `ingest_data.py` 和 `fetch_news.py` 脚本，将长篇大论切片 (Chunking) 转化为高维向量存储在 SQLite，通过 RAG 的精准召回直接作为模型的参考事实（Ground Truth）。两者互补，根治了时间滞后性。

**挑战二：复杂检索导致的系统高延迟**
如果在回答用户前，需要依次顺序拉取不同新闻源、查行情以及算向量，整个链路将耗时十几秒，用户体验极差。
*克服策略*：使用 FastAPI 的异步（Asyncio）特性重构 `chat_service` 的核心逻辑。我们将股票行情获取、外部新闻拉取、文本向量化三大 IO 密集型操作放到 `asyncio.gather` 中并发执行，把响应时间的瓶颈降到仅仅是最慢的那一个服务的响应时长，总延迟压缩至原来的 1/3 以下。

**挑战三：RAG 向量检索效率与本地部署的权衡**
引入庞大的向量数据库（如 Milvus / Chroma）会极大增加项目的部署负担与占用资源。
*克服策略*：降维打击，我们采用了最传统的 SQLite + 本地 `fastembed` 结合的方法。将生僻的向量序列化成 JSON 存在数据库长文本字段，并且在应用层手写了基于内存的余弦相似度（Cosine Similarity）算法，在保持百万级别字符知识库查询在百毫秒返回的同时，做到零数据库安装开销。

## 3. 结果（Results）

经过本地与远端接口测试，本项目达成了以下核心应用结果：

1. **金融知识图谱构建与检索表现**：通过 RAG 预处理与在线抓取配合，测试验证阶段显示，在面对诸如“根据最新总结，阿里2024的情况如何？”的问题时，系统能够完美规避原生大模型的凭空臆想，而是非常准时地提取出《9988.HK_阿里巴巴公司2024最新总结.txt》中提到的云业务增长与财务杠杆等本地细分论点。
2. **多模态与流畅渲染**：不仅提供了文本的深度交互，当用户查询具体的股票时，系统能在给出文本流式输出 (SSE Streaming) 的同时，由前端自动化渲染可视化的股票 K 线走势分析图，形成了完整的应用闭环。
3. **架构的高鲁棒性脱耦**：已彻底将项目与此前高学习门槛的废弃数据源解耦，全面切换向基于 Yahoo Finance（量化数字）与谷歌新闻（文字基本面）的通道。经过重构，极大提升了项目架构的健壮性。

## 4. 局限性和未来工作（Limitations and Future Work）

**主要局限性：**
1. **本地余弦库检索瓶颈**：当前手写的 SQLite 数组读取并作 JSON 反序列化解析的向量检索法，虽然轻量且容易部署，但一旦数据量突增，线性搜索必然导致内存溢出与高延迟。
2. **单轮指令逻辑**：当前模型的推理仍属于一次过（One-Shot），尚不支持复杂的 Agent 规划与反思能力。缺乏像 ReAct 框架那样主动去多次搜索、核实错误后在给出最终结论的能力。

**未来工作与迭代规划：**
1. **升级专业型数据库引擎**：拟全面切换至 PostgreSQL 结合 `pgvector` 扩展，用于代替 SQLite 计算向量库相似度，以突破段落级检索的算力瓶颈。
2. **多智能体 (Multi-Agent) 大一统编排**：计划引入 LangChain 框架，把当前的爬虫和行情源注册为真正的 Tools 函数。使底层大模型能够进行自主规划（自己决定是否继续调用查行情的工具），让它变成具有自修复与多步执行能力的智能体。

## 5. 工作分配（Work Allocation）

本项目采用敏捷协作的团队分工模式执行，各成员所负责的任务板块如下：

- **组员 A（后端架构与 LLM 核心）**：
  负责 FastAPI 主框架搭建与接口设计；实现 RAG 的大语言模型推理集成，包括编写 FastEmbed 本地向量化逻辑与余弦相似度算法。统筹调度的流式并发网络请求逻辑。
- **组员 B（数据源介入与工程化）**：
  负责外部 API 集成，包括从 YFinance 抓取行情数据、从 Google RSS 抓取实时网页新闻 (`live_news_service.py`)；并负责本地离线文件切块注入库的脚本 (`ingest_data.py`) 开发工作。
- **组员 C（前端可视化与交互体验）**：
  应用原生的 HTML5 / CSS3 与 Vanilla JavaScript，搭建平滑的流式 SSE 通信页面，并负责基于外部数据的股票 K 线走势图表前端无缝渲染及 UI 优化逻辑。
- **组员 D（数据库设计与测试部署）**：
  负责底层 SQLite 数据体系设计，包含用户注册身份认证模块和对历史聊天记忆状态的本地化储存管理；负责项目的整体 QA 功能测试统筹，以及最终的项目文档和报告的撰写维护。
